# 협업 컨벤션

## 1. 브랜치 전략
- `main`: 최종 통합 브랜치
- `feature/기능명`: 기능 개발
- `fix/버그명`: 버그 수정
- `docs/문서명`: 문서 작업
- `refactor/대상`: 리팩토링
- `test/대상`: 테스트 추가 또는 수정

### 브랜치 생성 예시
- `feature/login-api`
- `feature/backup-script`
- `fix/db-connection`
- `docs/readme-update`

## 2. 커밋 메시지 규칙
형식: `type: 설명`

### 사용 가능한 type
- `feat`: 새 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 수정
- `refactor`: 코드 구조 개선
- `test`: 테스트 추가 또는 수정
- `chore`: 환경 설정, 빌드 파일, 기타 작업

### 예시
- `feat: 재고관리 API 구현`
- `fix: PostgreSQL 연결 오류 수정`
- `docs: README 협업 규칙 추가`
- `test: 백업 스크립트 검증 테스트 추가`
- `chore: Docker 설정 정리`

### 커밋 작성 기준
- 한 커밋에는 한 가지 작업만 담기
- 너무 큰 작업은 여러 커밋으로 나누기
- 메시지는 짧고 구체적으로 작성하기
- 한글 작성 가능, 단 type은 통일하기

## 3. PR 작성 규칙
- PR은 기능 단위로 작게 올리기
- PR 제목은 `type: 내용 (#이슈번호)` 형식 권장
- PR 본문에는 작업 내용, 테스트 방법, 관련 이슈를 꼭 작성하기
- 리뷰어가 집중해서 봐야 할 부분이 있으면 명시하기
- UI 변경이 있으면 스크린샷 첨부하기

## 4. 문서 작성 규칙
- README에는 프로젝트 소개와 협업 규칙 요약만 작성
- 세부 규칙은 `docs/convention.md`에 정리
- 설계 문서, 배포 문서, 모니터링 문서는 `docs/` 폴더에 분리
- 회의록은 날짜별로 관리
- 트러블슈팅은 발생 원인, 해결 과정, 결과를 함께 작성

## 5. 협업 방식
- 매주 첫날: 이번 주 목표와 할 일 정리
- 매일 시작: 오늘 할 일과 담당자 확정
- 작업 중: 각자 로컬 또는 개인 AWS/VM 환경에서 먼저 검증
- 작업 완료 후: PR 생성 및 리뷰 요청
- 매일 마지막: 결과 공유, 문서화, 다음 날 할 일 정리
